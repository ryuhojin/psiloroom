# PSILO Agent Operating Guide

## 프로젝트 목적

`PSILO`는 그룹사 내 SI 프로젝트를 위한 일정 공유 및 협업 시스템이다.

핵심 목표는 다음과 같다.
- 달력 기반의 일정 공유 시스템을 제공한다.
- 일정, 채팅, 공지사항은 실시간으로 동기화된다.
- 프로젝트 내에서 개인 채팅, 그룹 채팅, 프로젝트 채팅이 가능해야 한다.
- 공지사항은 조직 단위와 프로젝트 단위로 실시간 전파되어야 한다.
- 사용자는 그룹사, 부서, 소속 역할 정보를 가진다.
- 외주 개발자는 그룹사와 부서에 `외주`로 표현하되 프로젝트 권한 범위 내에서만 기능을 사용한다.
- 프로젝트별로 기능 권한과 권한 레벨을 별도로 설정할 수 있어야 한다.
- 계정에는 그룹사 소속 외에 역할 소속(`고객사`, `수행사`, `외주개발사`)도 유지해야 한다.
- 배포는 Docker 기반으로 단순하고 재현 가능해야 한다.

현재 1차 범위는 아래를 우선한다.
- 관리자 웹
- 사용자 웹
- 인증/권한
- 실시간 일정
- 실시간 채팅
- 실시간 공지
- Docker 배포

후속 확장 범위는 아래와 같이 별도 관리한다.
- 칸반 기반 테스트 시스템
- 외주 개발자 평가 시스템

재사용 원칙은 다음과 같다.
- 같은 영역의 후속 작업은 가능한 한 기존 담당 에이전트가 계속 맡는다.
- 교차 영역 변경은 주 소유 에이전트와 영향받는 에이전트가 함께 검토한다.
- 메인 에이전트는 조정자 역할을 맡고 단독으로 설계 방향을 뒤집지 않는다.
- 설계 충돌 시 에이전트 1의 아키텍처 기준을 먼저 확인한다.
- 테스트 기준이 불명확할 경우 에이전트 6의 품질 게이트를 우선한다.

## 작업 규칙

전역 사용자 지침은 항상 우선 적용한다.

기본 작업 규칙은 다음과 같다.
- 코딩 전에 반드시 계획을 먼저 제시한다.
- 계획에는 `단계`, `변경 파일`, `테스트 방법`이 포함되어야 한다.
- 큰 변경은 실행 전에 승인받는다.
- 큰 변경의 기준은 의존성 추가, DB 스키마 변경, 인증/권한 정책 변경, 로깅 정책 변경, 배포 구조 변경이다.
- 민감정보는 채팅, 로그, 코드, 샘플 데이터에 남기지 않는다.
- 토큰, 비밀번호, 인증 헤더, 개인식별 정보는 필요 시 마스킹한다.
- 작업이 길어지면 중간 요약과 다음 할 일을 남긴다.
- 완료 시 변경 요약, 검증 방법, 남은 리스크 또는 후속 작업을 함께 남긴다.
- 모든 작업은 담당 서브 에이전트의 소유 범위를 우선 존중한다.
- 소유 범위를 넘는 수정은 영향 범위를 먼저 정리한 뒤 진행한다.
- 현재 범위에 없는 기능은 임의로 섞지 않는다.
- 후속 확장 가능성은 고려하되 MVP 범위를 흐리지 않는다.
- 실시간 기능은 일정, 채팅, 공지의 공통 이벤트 모델을 염두에 두고 설계한다.
- 프로젝트 권한은 전역 권한과 분리해서 관리한다.
- 외주 인력은 데이터 모델과 UI 모두에서 명확히 구분되어야 한다.
- 관리자 앱은 웹 기반을 기본으로 한다.
- 사용자 앱은 웹 우선으로 구현하고 Electron은 필요성이 검증될 때만 도입한다.
- Docker 배포 가능성은 초기 구조부터 유지한다.

## 작업 순서

작업은 아래 순서를 기본으로 한다.

1. 요구사항 정리
- 에이전트 1이 핵심 요구사항, 범위, 용어, 권한 모델, 데이터 모델 초안을 정리한다.
- 에이전트 6이 acceptance criteria와 초기 체크리스트 초안을 만든다.

2. 설계 확정
- 에이전트 1이 아키텍처, DB 스키마, 실시간 이벤트 구조를 확정한다.
- 에이전트 2, 3, 4, 5가 각자 화면/API/실시간 처리 설계를 구체화한다.

3. 구현 기반 구성
- 모노레포 구조, Docker, DB 마이그레이션, 공통 계약 타입, 공통 설정을 먼저 만든다.
- 인증/권한의 최소 골격을 먼저 구현한다.

4. 관리자 기능 구현
- 조직, 부서, 사용자, 프로젝트, 역할, 기능 권한 관리 기능을 우선 구현한다.
- 공지/일정/채팅 관리 기능은 관리자 정책과 연결되는 범위부터 구현한다.

5. 사용자 기능 구현
- 로그인, 워크스페이스 진입, 프로젝트 진입, Inbox를 먼저 구현한다.
- 이후 일정, 채팅, 공지, 알림, 읽음 처리, 실시간 동기화를 붙인다.

6. 통합 및 검증
- 에이전트 6이 통합 테스트, E2E, 실시간 시나리오, 권한 회귀를 검증한다.
- Docker 환경에서 기동, 재연결, 기본 운영 시나리오를 확인한다.

7. 마감 정리
- 변경 요약, 검증 결과, 잔여 리스크, 다음 단계 작업을 문서화한다.

## 승인 필요 항목

다음 항목은 작업 전에 반드시 승인받는다.
- 새 프레임워크 또는 대형 의존성 추가
- DB 스키마 신규 도입 또는 주요 변경
- 인증 방식 결정 또는 변경
- 권한 정책 구조 변경
- Redis, 메시지 브로커, 외부 알림 서비스 같은 인프라 추가
- 로깅/감사로그 정책 추가 또는 변경
- Electron 도입
- Docker Compose를 넘는 배포 구조 변경
- 외부 연동 시스템 도입
- 운영 환경 변수 구조 변경
- 데이터 보존 정책, 읽음 처리 정책, 공지 전파 정책 변경

승인 요청 시 반드시 아래를 함께 적는다.
- 변경 목적
- 영향 범위
- 변경 파일 또는 모듈
- 테스트 계획
- 예상 리스크
- 롤백 또는 대체 방안

## 커뮤니케이션 / 검증 원칙

커뮤니케이션 원칙은 다음과 같다.
- 작업 시작 전 현재 이해한 요구사항과 첫 단계 작업을 짧게 공유한다.
- 진행 중에는 현재 상태와 다음 행동을 짧게 알린다.
- 장시간 작업 시 중간 요약을 남긴다.
- 의사결정이 필요한 경우 선택지와 근거를 명확히 제시한다.
- 날짜, 버전, 환경 조건은 가능한 한 구체적으로 적는다.

검증 원칙은 다음과 같다.
- 구현 완료 주장은 검증 결과와 함께만 한다.
- 기능 검증은 단위 테스트, 통합 테스트, E2E, 수동 시나리오를 구분해서 기록한다.
- 실시간 기능은 재연결, 순서 보장, 읽음 처리, 권한 누락 여부를 반드시 확인한다.
- 권한 기능은 조직, 부서, 역할, 프로젝트 권한이 서로 섞이지 않는지 검증한다.
- 외주 인력 계정은 일반 계정과 분리된 시나리오로 검증한다.
- Docker 환경에서 최소 1회 이상 전체 기동 검증을 수행한다.
- 테스트를 못 돌린 경우에는 그 사실과 이유를 명시한다.
- 남은 리스크와 후속 작업은 숨기지 않고 명확히 남긴다.

## 운영 메모

이 문서는 프로젝트 진행 중 계속 유지한다.
역할, 소유 범위, 승인 기준이 바뀌면 먼저 이 문서를 갱신한 뒤 작업한다.
