# PSILO Checklist

## 1. 요구사항 추적

### 1.1 인증 / 조직 / 계정
- [ ] `RQ-AUTH-01` 그룹사별 로그인 방식이 정의되었다.
- [ ] `RQ-AUTH-02` 사용자 계정에 그룹사 정보가 필수로 연결된다.
- [ ] `RQ-AUTH-03` 사용자 계정에 부서 정보가 필수로 연결된다.
- [ ] `RQ-AUTH-04` 외주 개발자는 그룹사 또는 부서에 `외주` 식별값으로 구분 가능하다.
- [ ] `RQ-AUTH-05` 사용자 계정에 역할 소속(고객사, 수행사, 외주개발사 등)이 별도로 관리된다.
- [ ] `RQ-AUTH-06` 관리자와 일반 사용자의 접근 경계가 명확히 정의되었다.

### 1.2 프로젝트 / 권한
- [ ] `RQ-PERM-01` 프로젝트별 사용자 권한 레벨 부여가 가능하다.
- [ ] `RQ-PERM-02` 프로젝트별 기능 권한을 별도로 설정할 수 있다.
- [ ] `RQ-PERM-03` 권한 없는 사용자는 UI, API, 실시간 이벤트 모두 차단된다.
- [ ] `RQ-PERM-04` 권한 변경 사항은 세션 중에도 즉시 반영된다.
- [ ] `RQ-PERM-05` 외주 인력은 할당된 프로젝트 범위 내에서만 접근 가능하다.

### 1.3 일정 / 채팅 / 공지
- [ ] `RQ-CAL-01` 개인 일정 생성/조회/수정/삭제가 가능하다.
- [ ] `RQ-CAL-02` 그룹 일정 생성/조회/수정/삭제가 가능하다.
- [ ] `RQ-CAL-03` 프로젝트 일정 생성/조회/수정/삭제가 가능하다.
- [ ] `RQ-CAL-04` 일정 변경 사항이 권한 있는 사용자에게 실시간 반영된다.
- [ ] `RQ-CHAT-01` 개인 채팅이 가능하다.
- [ ] `RQ-CHAT-02` 그룹 채팅이 가능하다.
- [ ] `RQ-CHAT-03` 프로젝트 채팅이 가능하다.
- [ ] `RQ-CHAT-04` 채팅 메시지가 실시간 전달된다.
- [ ] `RQ-CHAT-05` 읽음/미읽음 상태가 정확히 유지된다.
- [ ] `RQ-NOTI-01` 공지사항을 실시간으로 전파할 수 있다.
- [ ] `RQ-NOTI-02` 공지 대상 범위(그룹사/프로젝트/역할)가 정확히 적용된다.
- [ ] `RQ-NOTI-03` 중요 공지에 대해 사용자 알림이 제공된다.

### 1.4 배포 / 운영
- [ ] `RQ-OPS-01` Docker 기반으로 개발/운영 환경을 일관되게 기동할 수 있다.
- [ ] `RQ-OPS-02` 재기동 후 데이터와 권한 설정이 유지된다.
- [ ] `RQ-OPS-03` 핵심 서비스 장애 시 오류 상태를 명확히 확인할 수 있다.

## 2. 구현 단계별 체크리스트

### 2.1 기획 / 설계
- [ ] 사용자 유형(관리자, 내부 사용자, 외주 사용자)이 정의되었다.
- [ ] 그룹사 / 부서 / 역할 소속 모델이 확정되었다.
- [ ] 프로젝트 권한 매트릭스가 문서화되었다.
- [ ] 일정 / 채팅 / 공지의 대상 범위 규칙이 정의되었다.
- [ ] 실시간 이벤트 목록과 이벤트 명세가 정리되었다.
- [ ] 예외 흐름과 실패 처리 정책이 정의되었다.

### 2.2 DB / 백엔드 기초
- [ ] 조직, 계정, 프로젝트, 권한 스키마가 확정되었다.
- [ ] 일정, 채팅, 공지, 알림, 읽음 상태 스키마가 확정되었다.
- [ ] 테넌시 경계 조건이 쿼리 레벨에서 반영되었다.
- [ ] 인증 / 인가 미들웨어 또는 정책 계층이 구현되었다.
- [ ] API 오류 코드와 응답 포맷이 표준화되었다.
- [ ] 감사 로그 또는 변경 이력 필요 범위가 정의되었다.

### 2.3 프론트엔드 기초
- [ ] 관리자 화면 라우팅과 권한 가드가 적용되었다.
- [ ] 사용자 화면 라우팅과 권한 가드가 적용되었다.
- [ ] 로그인 후 소속/권한 기반 초기 진입 흐름이 검증되었다.
- [ ] 일정, 채팅, 공지 UI의 빈 상태 / 로딩 상태 / 오류 상태가 정의되었다.
- [ ] 권한 없는 메뉴와 액션이 숨김 또는 차단 처리되었다.

### 2.4 실시간 기능 구현
- [ ] 일정 변경 실시간 반영이 동작한다.
- [ ] 채팅 메시지 실시간 송수신이 동작한다.
- [ ] 공지 실시간 전파가 동작한다.
- [ ] 읽음/미읽음 상태 실시간 동기화가 동작한다.
- [ ] 재연결 후 상태 복구 전략이 구현되었다.
- [ ] 중복 이벤트 및 순서 꼬임 방지 전략이 적용되었다.

### 2.5 통합 / 스테이징
- [ ] 관리자 시나리오 E2E가 작성되었다.
- [ ] 사용자 시나리오 E2E가 작성되었다.
- [ ] 외주 사용자 제한 시나리오 E2E가 작성되었다.
- [ ] 다중 사용자 동시 접속 시나리오가 검증되었다.
- [ ] Docker 기반 스테이징 기동 검증이 완료되었다.

## 3. 권한 / 테넌시 / 실시간 / 알림 테스트 포인트

### 3.1 권한
- [ ] UI 숨김만이 아니라 API 호출도 차단된다.
- [ ] 소켓 구독/이벤트 수신도 권한 기준으로 차단된다.
- [ ] 프로젝트별 권한 레벨 차이가 정확히 반영된다.
- [ ] 기능별 권한 변경 후 기존 세션에도 즉시 반영된다.
- [ ] 관리자 권한 오남용 경계가 확인되었다.

### 3.2 테넌시
- [ ] 그룹사 A 사용자가 그룹사 B 데이터를 조회할 수 없다.
- [ ] 그룹사 A 사용자가 그룹사 B 메타데이터도 간접적으로 볼 수 없다.
- [ ] 외주 사용자는 허용된 프로젝트에만 접근 가능하다.
- [ ] 고객사 / 수행사 / 외주개발사 역할 소속 조건이 정확히 적용된다.
- [ ] 검색, 자동완성, 최근 목록, 알림 배지에서도 타 테넌트 정보가 노출되지 않는다.

### 3.3 실시간
- [ ] 소켓 연결 성공 / 실패 / 재연결 흐름이 검증되었다.
- [ ] 이벤트 중복 수신이 방지된다.
- [ ] 이벤트 누락 시 복구 또는 재동기화가 가능하다.
- [ ] 메시지/일정 변경 순서가 일관되게 보장되거나 보정된다.
- [ ] 동시 수정 충돌 시 사용자에게 명확한 결과가 제공된다.
- [ ] 권한 제거 직후 더 이상 실시간 이벤트를 수신하지 않는다.

### 3.4 알림
- [ ] 일정, 채팅, 공지 알림 우선순위가 정의되었다.
- [ ] 동일 이벤트에 대한 중복 알림이 방지된다.
- [ ] 읽음/미읽음 상태가 다중 기기에서 일관된다.
- [ ] 오프라인 또는 연결 해제 후 복구 시 미수신 항목이 동기화된다.
- [ ] 브라우저/PC 알림 권한 미허용 시 대체 UX가 제공된다.

## 4. Docker 배포 검증 항목

- [ ] `docker compose up`으로 전체 서비스가 정상 기동한다.
- [ ] 앱, DB, 캐시, 메시징, 프록시의 헬스체크가 통과한다.
- [ ] 필수 환경 변수가 없으면 안전하게 실패하고 원인을 표시한다.
- [ ] 마이그레이션이 초기 기동 시 정상 수행된다.
- [ ] 시드 데이터가 필요 시 멱등적으로 적용된다.
- [ ] 볼륨 마운트 후 재기동 시 데이터가 유지된다.
- [ ] 컨테이너 간 네트워크 연결과 서비스 이름 해석이 정상이다.
- [ ] 시간대 설정이 일관되어 일정/공지 시간이 어긋나지 않는다.
- [ ] 로그가 표준 출력으로 수집 가능하다.
- [ ] 로그에 비밀번호, 토큰, 인증 헤더 등 민감정보가 남지 않는다.
- [ ] 단일 컨테이너 재시작 시 나머지 서비스가 정상 복구 또는 재연결된다.
- [ ] 로컬 개발용과 운영용 Compose 구성이 구분되어 있다.

## 5. 릴리스 품질 게이트

- [ ] P1 결함이 0건이다.
- [ ] P2 결함이 릴리스 허용 기준 이하이다.
- [ ] 핵심 요구사항 추적 항목이 모두 완료 상태이다.
- [ ] 권한 / 테넌시 회귀 테스트가 모두 통과했다.
- [ ] 일정 / 채팅 / 공지 실시간 시나리오가 모두 통과했다.
- [ ] 관리자 핵심 E2E 시나리오가 통과했다.
- [ ] 사용자 핵심 E2E 시나리오가 통과했다.
- [ ] 외주 사용자 제한 시나리오가 통과했다.
- [ ] Docker 배포 스모크 테스트가 통과했다.
- [ ] 운영 환경 변수, 시크릿, 로그 정책 검토가 완료되었다.
- [ ] 릴리스 노트와 알려진 제한사항이 정리되었다.
- [ ] 회귀 테스트 결과와 배포 승인 기록이 남아 있다.
